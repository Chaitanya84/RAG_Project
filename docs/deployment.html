🚀 7. Deployment on Render

This section explains how the app is deployed to Render — a cloud hosting platform — so anyone with the URL can use it without installing anything.

💡 What is Render? A cloud platform (like Heroku) where you push your code and they run it on a server 24/7. You get a public URL like https://your-app.onrender.com. The free tier gives 512 MB RAM and spins down after 15 minutes of inactivity.

The Three Deployment Files

1. Procfile — The Start Command

web: streamlit run app.py --server.port=$PORT --server.address=0.0.0.0
  • web: — Tells Render this is a web service (not a background worker).
  • streamlit run app.py — The command to start the app.
  • --server.port=$PORT — Render assigns a random port via the $PORT environment variable. Streamlit must listen on it.
  • --server.address=0.0.0.0 — Listen on all network interfaces (required so Render's proxy can reach the app).
💡 What is 0.0.0.0? By default, Streamlit only listens on 127.0.0.1 (localhost — only your machine). 0.0.0.0 means "accept connections from any IP address", which is necessary on a server because Render's load balancer connects from a different internal IP.

2. runtime.txt — Python Version

python-3.12.3

Tells Render exactly which Python version to install. This must match the version you developed with locally.

3. .streamlit/config.toml — Server Settings

[server]
maxUploadSize = 200        # Allow uploading PDFs up to 200 MB
headless = true            # Don't try to open a browser (no GUI on a server)
address = "0.0.0.0"        # Same as Procfile — listen on all interfaces
enableXsrfProtection = false  # Needed for OAuth redirects to work
maxMessageSize = 200       # Max WebSocket message size in MB

[browser]
gatherUsageStats = false   # Don't send telemetry to Streamlit

[theme]
primaryColor = "#4285F4"   # Google blue — matches the login button
⚠️ Why disable XSRF protection? Cross-Site Request Forgery (XSRF) protection blocks POST requests from other domains. But Google OAuth sends the user back with a POST-like redirect that can trigger this protection. Disabling it is a pragmatic trade-off for a single-user/demo app. For a production app with many users, you'd implement proper CSRF tokens instead.

Environment Variables on Render

Since .env and client_secret.json are gitignored (they contain secrets), you must set them as environment variables in the Render dashboard:

VariableValueNotes
OPENAI_API_KEYsk-proj-...Your OpenAI API key
OPENAI_EMBEDDING_MODELtext-embedding-3-smallOptional (has default)
OPENAI_CHAT_MODELgpt-4o-miniOptional (has default)
REDIRECT_URIhttps://your-app.onrender.com/Must match Google OAuth config
GOOGLE_CLIENT_SECRET_JSON{"web":{"client_id":"..."}}The entire contents of client_secret.json as a string
DEBUGfalseDisable debug logging in production
💡 How GOOGLE_CLIENT_SECRET_JSON works: Since client_secret.json isn't in the repo, config.py checks if the file exists on startup. If it doesn't (on Render), it reads the GOOGLE_CLIENT_SECRET_JSON env var and creates the file on disk at runtime. Clever trick to avoid committing secrets.

Step-by-Step Render Deployment

1 Push Code to GitHub

git add .
git commit -m "Ready for deployment"
git push origin main

2 Create a Render Web Service

  1. Go to render.com and sign up
  2. Click "New +" → "Web Service"
  3. Connect your GitHub repo
  4. Set Root Directory to RAG_Streamlit_App
  5. Set Build Command to pip install -r requirements.txt
  6. Set Start Command to streamlit run app.py --server.port=$PORT --server.address=0.0.0.0 (or leave blank to use Procfile)
  7. Add all environment variables from the table above
  8. Click "Create Web Service"

3 Update Google OAuth Redirect

In the Google Cloud Console, add your Render URL as an authorized redirect URI:

https://your-app.onrender.com/

This must exactly match the REDIRECT_URI environment variable on Render.

Memory Optimization for Render (512 MB)

The free tier of Render only gives 512 MB of RAM. The project has several strategies to stay within this limit:

StrategyWhereWhat It Does
Generator-based PDF readingembedandchunk.pyReads one page at a time instead of loading entire PDF
Regex instead of spaCyembedandchunk.pySaves ~150 MB by not loading an NLP model
Streaming CSV writesembedandchunk.pyWrites rows to disk immediately
gc.collect()embedandchunk.pyForces Python garbage collector between phases
Background threadmainPage.pyProcessing runs in a thread so the UI stays responsive
@st.cache_resourcechatPage.py, quizPage.pyLoads embeddings only once, not on every page re-run
