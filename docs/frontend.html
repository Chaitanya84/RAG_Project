🖥️ 6. Streamlit Frontend

The app has four pages built with Streamlit. This section explains how each page works — every widget, callback, and session state variable.

💡 What is Streamlit? A Python framework that turns scripts into web apps. You write normal Python code with calls like st.title("Hello") or st.button("Click me"), and Streamlit renders them as HTML elements in your browser. No HTML/CSS/JS needed. Every time a user interacts with a widget, Streamlit re-runs the entire script from top to bottom.

How Streamlit Multi-Page Apps Work

  • The entry point is app.py (run with streamlit run app.py).
  • Any .py file inside a pages/ folder automatically becomes a sidebar page.
  • st.set_page_config() is called only once in app.py — never in the page files.
  • st.session_state is a dictionary that persists across re-runs within a single browser session. It's how pages share data.

🔐 Page 1: Login — app.py

This is the first thing users see. It handles Google OAuth authentication.

What happens step by step:

  1. User sees a "Login with Google" button (rendered as an HTML link).
  2. Clicking it redirects to Google's login page via build_auth_url().
  3. After the user signs in, Google redirects back to the app with a ?code=... query parameter.
  4. exchange_code_for_tokens() sends that code to Google and gets back an ID token.
  5. The ID token is verified with id_token.verify_oauth2_token().
  6. The user's email is stored in st.session_state.user_email.
# Build the Google login URL
def build_auth_url():
    params = {
        "client_id": CLIENT_ID,
        "redirect_uri": REDIRECT_URI,   # http://localhost:8501/
        "response_type": "code",
        "scope": "openid email profile",
    }
    return f"{AUTH_URI}?{urllib.parse.urlencode(params)}"

📂 Page 2: Upload & Process — pages/mainPage.py

This page lets users upload PDFs and processes them into a knowledge base.

Widgets used:

Checkboxst.checkbox("Force re-process")Deletes previous outputs before re-processing
Buttonst.button("🚀 Process All PDFs")Starts the background processing thread

Background Processing (the clever part):

PDF processing takes 30–120 seconds. If we did it in the main Streamlit thread, the UI would freeze. Instead, we use a background thread:

# 1. Save uploaded files to disk
for uf in uploaded_files:
    with open(os.path.join(UPLOAD_DIR, uf.name), "wb") as f:
        f.write(uf.getbuffer())

# 2. Start processing in a background thread
thread = threading.Thread(
    target=_run_processing_in_background,
    args=(UPLOAD_DIR, pages_csv, chunks_csv, embeddings_pkl),
    daemon=True,
)
thread.start()

# 3. Page auto-refreshes every 3 seconds to check status
time.sleep(3)
st.rerun()
⚠️ Why file-based signaling? Background threads in Streamlit cannot safely write to st.session_state (it causes ScriptRunContext errors). So the thread writes status to a JSON file on disk (_processing_status.json), and the main thread reads it every 3 seconds: 
# Thread writes:  {"status": "done", "processed_files": ["book.pdf"]}
# Main thread reads and updates session_state

💬 Page 3: Chat — pages/chatPage.py

This page provides a ChatGPT-like interface for asking questions about uploaded PDFs.

How it works:

# 1. Load knowledge base (cached — only loaded once)
@st.cache_resource
def get_rag_resources(_chunks_csv, _embeddings_path):
    return load_rag_resources(chunks_csv=_chunks_csv, embeddings_path=_embeddings_path)

pages_and_chunks, embeddings = get_rag_resources(chunks_csv, embeddings_path)

# 2. Display chat history (previous messages)
for msg in st.session_state.chat_history:
    with st.chat_message(msg["role"]):
        st.markdown(msg["content"])

# 3. Native chat input bar (pinned to bottom)
user_input = st.chat_input("Ask a question about the document…")

# 4. On submit: get RAG answer and display it
if user_input:
    answer = answer_with_rag_cached(user_input, pages_and_chunks, embeddings)
    st.session_state.chat_history.append({"role": "assistant", "content": answer})
💡 What is @st.cache_resource? A Streamlit decorator that caches the return value of a function. The first time it's called, it runs normally and stores the result. On every subsequent re-run (every user interaction!), it returns the cached result instantly. This means the chunks CSV and embeddings are loaded from disk only once, not on every keystroke.

🧠 Page 4: Quiz — pages/quizPage.py

This page auto-generates multiple-choice quizzes from the uploaded PDFs.

Quiz Configuration Widgets:

WidgetOptionsPurpose
st.selectbox("Quiz Source")All documents, specific file, or random passagesChoose where questions come from
st.selectbox("Difficulty")Easy, Medium, HardControls question complexity
st.selectbox("Questions")3, 5, 7, 10How many questions to generate

Quiz Lifecycle:

  1. Generate: User clicks "Generate Quiz" → chunks are sampled → sent to ChatGPT → MCQ JSON returned → stored in st.session_state.quiz_questions.
  2. Take: Questions rendered inside st.form() with st.radio() for each. User selects answers and clicks "Submit".
  3. Results: Score calculated, shown with progress bar. Each question reviewed with ✅/❌, correct answers highlighted in green, wrong in red, and explanations in expandable sections.
# Rendering one quiz question
with st.form("quiz_form"):
    for i, q in enumerate(questions):
        st.markdown(f"**Q{i+1}.** {q['question']}")
        choices = [f"{key}: {val}" for key, val in q["options"].items()]
        st.radio(f"Answer Q{i+1}:", options=choices, key=f"q_{i}")
    st.form_submit_button("✅ Submit Answers")
💡 Why st.form()? Normally, Streamlit re-runs the script every time any widget changes. With a form, changes are batched — the script only re-runs when the submit button is clicked. This prevents the quiz from resetting every time the user selects a radio button.
